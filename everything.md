# Blood Bank Finder: Full Codebase Documentation

This document provides a comprehensive explanation of every file in the Blood Bank Finder project, detailing their purpose, importance, and specific contribution to the application.

---

## Project Overview
The **Blood Bank Finder** is a multi-platform application (Android, iOS, Web) designed to bridge the gap between blood donors and seekers/hospitals. It features a Flutter-based mobile/web interface and a FastAPI-powered backend with Firebase as the primary data store and authentication provider.

---

## 📱 Frontend: Flutter Application (`lib/`)

The frontend is built using Flutter, following a feature-based architecture combined with service-oriented patterns.

### Core Configuration
- **[main.dart](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/lib/main.dart)**: The entry point of the Flutter application. It initializes Firebase, sets up global state management using `MultiProvider`, and defines the base `MaterialApp` with global themes and the initial splash screen.
- **[firebase_options.dart](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/lib/firebase_options.dart)**: Automatically generated by the Firebase CLI. It contains platform-specific (Android, iOS, Web) configuration keys required to connect the Flutter app to the correct Firebase project.

### Core Logic (`lib/core/`)
- **`core/providers/auth_provider.dart`**: Manages the authentication state system-wide. It handles login, logout, and registration logic, exposing current user data to the entire app.
- **`core/theme/app_theme.dart`**: Defines the visual identity of the app. It centralizes colors, typography, and button styles to ensure UI consistency.

### Data Models (`lib/models/`)
These files define the structure of data exchanged within the app and with Firebase/FastAPI.
- **`blood_request_model.dart`**: Structure for blood requests, including blood type, urgency, location, and status.
- **`hospital_model.dart`**: Data structure for hospital details (name, contact, location, assigned admins).
- **`inventory_model.dart`**: Manages blood stock levels for various types (A+, B-, etc.) within a hospital.
- **`user_model.dart`**: Structure for user profiles (donors, patients, or admins), including their blood type and history.
- **`notification_model.dart`**: Handles the structure for alerts and messages sent to users.

### Features (`lib/features/`)
The app is modularized by user roles and specific functionalities.
- **`features/auth/`**: Contains screens for login, signup, and splash logic.
- **`features/user/`**:
    - `request_blood_screen.dart`: UI for seekers to post new requests.
    - `profile_screen.dart`: View and edit personal donor profile.
- **`features/hospital/`**:
    - `hospital_admin_dashboard.dart`: The hub for hospital administrators to manage stock and view requests.
- **`features/super_admin/`**: High-level screens for system administrators to manage hospitals and overall system health.

### Services (`lib/services/`)
- **`database_service.dart`**: The primary data access layer. It contains functions for CRUD operations (Create, Read, Update, Delete) interacting directly with Firebase Firestore.

## ⚙️ Backend: FastAPI (`backend/`)

The backend provides a RESTful API layer to handle complex logic, notifications, and centralized data management.

### Application Logic (`backend/app/`)
- **[main.py](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/backend/app/main.py)**: The FastAPI entry point. It registers all API routers and defines the base infrastructure for the web server.
- **[config.py](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/backend/app/config.py)**: Manages environment variables and initializes the Firebase Admin SDK connection.
- **`models.py`**: Defines Pydantic schemas for request/response validation, ensuring the API receives and sends data in the correct format.

### Routers (`backend/app/routers/`)
Modularized API endpoints:
- **`users.py`**: Endpoints for user profile management.
- **`hospitals.py`**: API logic for managing hospital registries.
- **`inventory.py`**: Handles blood stock updates and synchronization.
- **`requests.py`**: Logic for matching blood requests with available donors/hospitals.
- **`notifications.py`**: Logic for triggering push notifications via Firebase Cloud Messaging (FCM).

## � Real-World Operational Examples

To understand how these files collaborate, here are two critical workflows:

### Scenario 1: A User Requests Blood
1.  **UI Entry**: The user opens **[request_blood_screen.dart](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/lib/features/user/screens/request_blood_screen.dart)** and fills out the form.
2.  **Data Creation**: A **`BloodRequestModel`** is instantiated with the user's ID, blood type, and target hospital.
3.  **Database Write**: The **`DatabaseService.createBloodRequest()`** method is called, which pushes a new document into the Firestore `blood_requests` collection.
4.  **Admin Visibility**: The hospital admin's dashboard uses **`DatabaseService.streamHospitalRequests()`** to listen for new requests in real-time.
5.  **Status Update**: When the admin approves the request, **`DatabaseService.updateRequestStatusWithNotification()`** is triggered.
6.  **User Alert**: This method updates the request status and simultaneously creates a **`NotificationModel`** in the `notifications` collection, which the user sees immediately on their device.

### Scenario 2: Hospital Admin Updates Blood Inventory
1.  **Direct Update (App)**: In the **`hospital_admin_dashboard.dart`**, an admin modifies the units for "O+".
2.  **Service Call**: The app calls **`DatabaseService.updateInventory()`**.
3.  **API Sync (Backend)**: If using the backend route, the app hits the **`inventory.py`** FastAPI endpoint (PUT `{hospital_id}/inventory/{blood_type}`).
4.  **Backend Logic**: The FastAPI router uses **`config.py`** to get a database connection and calls **`FirestoreService.update_inventory()`** (on the backend) to update the exact same Firestore collection.
5.  **Reflection**: All users viewing that hospital's stock will see the updated count immediately because they are listening to a stream from **`DatabaseService.streamInventory()`**.

- **[pubspec.yaml](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/pubspec.yaml)**: Defines Flutter dependencies (Firebase, Provider, Google Fonts, etc.) and asset paths.

- **[firestore.rules](file:///c:/Users/The%20Chosen%20One/OneDrive/Documents/blood%20bank%20finder/firestore.rules)**: Defines the security settings for the Firestore database, controlling who can read/write specific data collections.
- **`.gitignore`**: Ensures that sensitive files (like API keys, `.env`, and `serviceAccountKey.json`) are not committed to version control.
