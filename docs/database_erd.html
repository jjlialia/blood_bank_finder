Relationships & Cardinality

User ↔ BloodRequest (1:N)
A single user can initiate multiple blood requests or donations over time. Each request is linked to exactly one user.
Hospital ↔ BloodRequest (1:N)
A hospital handles many requests from different users. Each request is targeted at a specific hospital.
Hospital ↔ Inventory (1:N)
Each hospital maintains its own inventory levels. In Firestore, this is implemented as a sub-collection for optimal scoping.
User ↔ ChatRoom (N:M)
Users can participate in multiple chats, and each chat has multiple participants (stored as an array of UIDs).
ChatRoom ↔ Message (1:N)
A chat room contains an ordered stream of messages. Messages are scoped to a specific room ID.
